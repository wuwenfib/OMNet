一、整体架构

OMNeT++ 由哪几层组成

OMNeT++ 是一个基于组件的离散事件仿真框架。整个系统从下到上分为四层：

┌─────────────────────────────────────────────────────┐ │ 用户仿真模型层 │ │ NED 文件（拓扑描述） + C++ 模块代码（行为逻辑） │ ├─────────────────────────────────────────────────────┤ │ 仿真内核层 │ │ 事件队列 · 仿真时钟 · 消息路由 · 内存管理 │ ├─────────────────────────────────────────────────────┤ │ 运行环境层（Env） │ │ Cmdenv（命令行） Qtenv（图形界面） │ ├─────────────────────────────────────────────────────┤ │ 支撑工具层 │ │ opp_makemake opp_msgc opp_run IDE 插件 │ └─────────────────────────────────────────────────────┘
用户仿真模型层

开发者编写的 NED 文件和 C++ 模块，描述"网络长什么样"和"节点怎么行动"。

仿真内核层

OMNeT++ 框架核心，负责驱动事件队列、推进仿真时钟、在模块间传递消息。

运行环境层

Cmdenv 用于批量脚本运行，Qtenv 提供可暂停、可可视化的图形调试界面。

支撑工具层

opp_msgc 编译 .msg 文件，opp_makemake 生成 Makefile，opp_run 启动仿真。

二、核心概念

模块、门、信道、消息
┌──────────────────────────────┐ │ 复合模块 Network │ │ │ │ ┌──────────┐ ┌─────────┐ │ │ │ 简单模块 │ │ 简单模块 │ │ │ │ NodeA │ │ NodeB │ │ │ │ │ │ │ │ │ │ gate ○─────────○ gate│ │ │ └──────────┘ └─────────┘ │ │ ↑ 信道（delay=100ms） │ └──────────────────────────────┘
概念 NED 关键字 C++ 基类 说明
简单模块 simple cSimpleModule 有 C++ 代码实现行为的最小执行单元
复合模块 module cModule 由子模块拼装而成，自身无 C++ 行为代码
网络 network — 顶层复合模块，仿真的入口
门（Gate） input / output / inout cGate 模块的连接接口，消息从门进出
信道（Channel） { delay=… } cChannel 连线上的属性，可设延迟、带宽、误码率
消息（Message） packet / message (.msg) cMessage / cPacket 模块间传递的数据单元，也用作定时器
💡
简单模块 vs 复合模块：简单模块是叶子节点，复合模块是容器。实际仿真行为只发生在简单模块里。复合模块只是用来组织拓扑结构。

三、NED 语言

描述网络拓扑的专用语言

NED（Network Description）是 OMNeT++ 专用的拓扑描述语言，负责"画网络图"，C++ 代码负责"写行为逻辑"，两者通过模块名绑定。

// 1. 声明简单模块（描述接口，不写行为）
simple Router
{
    parameters:
        int    myAddr   = default(1);   // NED 参数，C++ 用 par() 读取
        double delay    = default(0.1);
    gates:
        input  in;        // 单向输入门
        output out[];     // 动态大小输出门阵列
}

// 2. 声明网络（描述拓扑）
network MyNetwork
{
    submodules:
        router : Router { myAddr = 5; }   // 实例化并赋参数
        host0  : Host;
        host1  : Host;
    connections:
        host0.out  --> { delay = 10ms; } --> router.in;  // 带信道参数的连线
        router.out++ --> host0.in;   // ++ 动态追加门阵列元素
        router.out++ --> host1.in;
}
🔗
NED 与 C++ 的绑定：NED 里写 simple Router，C++ 里写 class Router : public cSimpleModule 和 Define_Module(Router)，框架靠模块名字符串自动匹配。
⚙️

四、离散事件仿真引擎

事件队列如何驱动仿真时钟

OMNeT++ 是离散事件仿真（Discrete Event Simulation，DES）框架。仿真时间不是连续流逝的，而是由事件驱动跳跃推进的。

事件队列（按仿真时间戳从小到大排序） ┌──────────┬──────────────────────────────────────┐ │ 时间戳 │ 事件 │ ├──────────┼──────────────────────────────────────┤ │ t = 0.0s │ nodeA 的 "start" 自消息触发 │ │ t = 0.1s │ nodeB 收到 "ping"（经过 100ms 信道延迟）│ │ t = 1.1s │ nodeB 的 "reply-timer" 触发 │ │ t = 1.2s │ nodeA 收到 "pong" │ │ t = 2.2s │ nodeA 的 "reply-timer" 触发 │ │ ... │ ... │ └──────────┴──────────────────────────────────────┘ 引擎主循环： while (队列不空 && 未到 sim-time-limit) { 取出队头事件（最小时间戳） 把仿真时钟 跳到 该时间戳 ← 不是等待，是直接跳 调用对应模块的 handleMessage() }
1
initialize() 阶段 所有模块按层次依次调用 initialize()，模块在此安排初始事件（如 scheduleAt(0, msg)）。
2
事件循环阶段 引擎反复从队列取最早事件，推进仿真时钟，调用 handleMessage()。模块在此安排后续事件或发送消息。
3
消息传递 send(msg, gate) 不是"立即送达"，而是把消息放入队列，时间戳 = 当前时间 + 信道延迟。
4
自消息（定时器） scheduleAt(t, msg) 把消息放入队列指向自己，时间戳 = t。这是 OMNeT++ 里实现"等待"的唯一正确方式，不能用 sleep()。
5
仿真结束 到达 omnetpp.ini 里的 sim-time-limit，或事件队列为空，引擎调用 finish() 后退出。
⚠️
关键认知：仿真时间与真实时间无关。waitTime = 100s 和 waitTime = 1s 的运行速度几乎相同，因为引擎只是跳跃时间戳，不做真实等待。
✉️

五、消息生命周期与内存管理

谁创建，谁负责，何时释放

OMNeT++ 采用手动内存管理规则，消息对象在堆上分配，框架不自动回收。

操作 所有权变化 你需要做什么
new cMessage("x") 你拥有 负责最终 delete
send(msg, gate) 所有权交给框架 发出后不能 delete，也不能再访问
scheduleAt(t, msg) 所有权交给框架 触发后框架自动交回，在 handleMessage 里负责 delete
handleMessage(msg) 接收 所有权交回你 处理完必须 delete（或 send 转发出去）
cancelAndDelete(msg) — 取消未触发的自消息并释放，最常用的取消方式 
void Router::handleMessage(cMessage *msg) {
    AppPacket *pkt = check_and_cast<AppPacket*>(msg);  // 安全转型

    if (pkt->getDestAddr() == myAddr) {
        delete pkt;          // 到达终点，消费并释放 ✓

    } else {
        auto it = routingTable.find(pkt->getDestAddr());
        if (it != routingTable.end()) {
            send(pkt, "out", it->second);  // 转发，所有权交出 ✓
        } else {
            delete pkt;        // 丢包，必须手动释放 ✓
        }
    }
}

┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │ 写 NED │ → │ 写 .msg │ → │ 写 C++ │ → │ 配置 ini │ │ 描述拓扑 │ │ 定义消息 │ │ 实现行为 │ │ 设置参数 │ └──────────┘ └──────────┘ └──────────┘ └──────────┘ │ │ opp_msgc │ 自动生成 │ *_m.h / *_m.cc │ │ │ ┌──────────────────────────────────────────────────────┐ │ opp_makemake -f --deep -o <名称> 生成 Makefile │ │ make -j$(nproc) 编译链接 │ │ ./out/gcc-release/<名称> -u Cmdenv omnetpp.ini 运行 │ └──────────────────────────────────────────────────────┘
工具 作用
opp_msgc 将 .msg 文件编译成 C++ 类（*_m.h + *_m.cc），make 时自动调用
opp_makemake 扫描目录生成适合 OMNeT++ 的 Makefile，支持 --deep 递归子目录
Cmdenv 命令行运行环境，适合批量仿真和查看文本日志
Qtenv 图形运行环境，可动画展示消息流动、单步调试、查看模块内部状态
